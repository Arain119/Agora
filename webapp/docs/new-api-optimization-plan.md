# 基于 `QuantumNous/new-api` 的 Agora 优化方案（深度借鉴版）

> 目标：在保持“轻量、OpenAI 兼容、NVIDIA 聚合”定位的前提下，借鉴 `new-api` 的工程化能力，补齐稳定性、可扩展性与运营能力短板。

## 1. 现状诊断（Agora）

从代码可以看到当前版本具备可用的最小闭环：

- 已有 OpenAI 兼容端点（`/v1/models`、`/v1/chat/completions`、`/v1/completions`）。
- 有多 Key 轮询、每 Key RPM 限流、User Token 限流。
- 有简单管理后台（keys/tokens/logs/password/stats）。

但也存在影响生产可用性的关键问题：

1. **密钥与密码硬编码在源码内**（极高风险）
   - 默认 NVIDIA Key、默认用户 token、默认 admin 密码均在内存初始化中定义。
2. **状态仅在内存中**
   - Worker 实例重启、扩缩容、冷启动后状态不一致或丢失。
3. **路由策略仅“当前分钟使用率最低”**
   - 未纳入失败率、错误类型、延迟、半开熔断等维度，易在上游异常时放大故障。
4. **鉴权模型过于简化**
   - admin token 直接等于密码、没有会话/过期机制、没有操作审计。
5. **可观测性不足**
   - 仅有近 500 条内存日志，无法做跨实例追踪、告警、容量评估。

## 2. 借鉴 `new-api` 的方向（抽象能力，不照搬实现）

从 `new-api` 仓库公开信息可归纳出三类能力值得借鉴：

1. **“网关化”能力**：统一入口、多模型/多协议兼容、管理面与控制面分离。
2. **“可运营”能力**：文档、部署方式、管理界面、配置化与长期运维友好。
3. **“可扩展”能力**：在单一核心上游下，围绕多密钥、多区域、策略化调度与治理能力做纵向扩展。

对 Agora 的启发是：
- 把“能跑”升级为“可控、可观测、可扩展”。
- 保持“固定 NVIDIA Provider”不变，把演进重点放在高可用、高安全、可观测和可运营。

## 3. 优化蓝图（按优先级分层）

## P0（1~2 周，必须优先）

### P0-1 安全基线

- 移除硬编码密钥/密码/默认 token。
- 统一由环境变量 + Secret Store 注入（Cloudflare Secrets/KV/DO）。
- 管理接口增加：
  - 密码哈希存储（Argon2/bcrypt）。
  - 登录签发短期 JWT（含过期时间）。
  - 高危操作二次校验（修改密码、删除 key、导出 token）。

**收益**：快速降低密钥泄露和后台被接管风险。

### P0-2 持久化状态

- 将 `keys` / `tokens` / `password` / `stats` 从内存迁移到 D1（或 PostgreSQL）。
- 日志从“仅内存 500 条”升级为可分页存储（最少保留 7~30 天）。
- 内存层仅保留热缓存，并设置版本号与缓存失效策略。

**收益**：解决重启丢数据、实例不一致、容量不可评估问题。

### P0-3 路由容错

- 在现有“使用率”策略上增加多维评分：
  - `score = rpm_usage * w1 + fail_rate * w2 + p95_latency * w3 + cooldown_penalty`
- 增加熔断器：连续失败阈值触发 open，冷却后 half-open 探测。
- 支持按错误码退避（429/5xx 分开处理）。

**收益**：上游抖动时成功率更稳、尾延迟更低。

## P1（2~4 周，建议跟进）

### P1-1 NVIDIA 专用适配层（固定 Provider）

- 保持单一上游策略：仅 NVIDIA Provider。
- 将现有转发逻辑下沉为 `NvidiaProviderService`，并按能力拆分为：
  - `listModels` / `chat` / `completion` / `stream`
- 扩展点只允许在 NVIDIA 范围内演进（如多区域端点、模型能力标签、故障域隔离），不引入其他 Provider。

**收益**：在不增加多上游复杂度的前提下，显著提升代码可维护性与稳定性。

### P1-2 统一请求规范化与参数映射

- 在 `model-adapter` 基础上增加：
  - 参数白名单、黑名单、模型能力检查。
  - 不兼容参数自动降级（并记录 warning）。
  - 可选“严格模式”：参数不兼容直接 400。

**收益**：降低模型差异导致的错误率，提高 SDK 兼容性。

### P1-3 令牌体系升级

- Token 支持标签、项目、用途、到期时间、IP 白名单。
- 限流支持层级：全局 -> 组织 -> 项目 -> token。
- 增加配额（按日/月请求数或 token 数）与超额告警。

**收益**：从“个人工具”走向“团队可运营平台”。

## P2（4~8 周，可作为增长项）

### P2-1 可观测性平台化

- 指标：QPS、成功率、P50/P95/P99、上游错误码分布、各 key 健康分。
- 日志：接入结构化日志（JSON），关联 `request_id` / `trace_id`。
- 告警：连续 5xx、成本异常、单 key 故障自动通知。

### P2-2 成本与治理

- 按模型维护成本系数，支持成本看板。
- 加入“路由策略模板”：
  - 低成本优先
  - 低延迟优先
  - 高稳定优先
- 设定审计日志与操作留痕，满足企业合规。

### P2-3 管理台体验

- Dashboard 增加趋势图（分钟/小时维度）。
- Key 详情页增加健康时间线。
- Token 维度统计（调用量、成功率、热点模型）。

## 4. 针对当前代码的“具体落地清单”

1. `src/lib/state.ts`
   - 删除硬编码密钥与默认敏感信息，改为从运行时配置注入。
   - 将状态读写抽象为 Repository（memory + d1 两套实现）。
2. `src/index.tsx`
   - 在转发前后注入 request_id。
   - 对 429 / 5xx 加入重试与退避策略（可配置最大重试次数）。
3. `src/routes/admin.ts`
   - 登录改 JWT，admin 密码改哈希验证。
   - 所有写操作记录审计日志（who/when/what）。
4. `src/lib/models.ts` + `src/lib/model-adapter.ts`
   - 增加 model capability schema，做参数级兼容检查。

## 5. 建议的里程碑（可直接进迭代）

- **Milestone A（安全与稳定）**：P0 全部完成。
- **Milestone B（NVIDIA 专用架构稳态）**：完成 NVIDIA 专用适配层 + 参数规范化。
- **Milestone C（运营化）**：观测与成本治理上线。

## 6. 关键 KPI（优化是否成功）

- 7 日成功率：`>= 99.5%`
- P95 延迟：下降 `20%+`
- 上游故障恢复时间（MTTR）：下降 `30%+`
- 密钥泄露风险项：从“高”降为“低”（无硬编码密钥）
- 管理操作可追溯率：`100%`
